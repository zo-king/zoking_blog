# ADR-0001: 采用后台动态管理 + Hugo 静态发布架构

## 状态

Proposed

## 背景

项目目标是构建成熟完整的全栈博客系统，同时保留 Hugo Theme Stack 的 C 端页面设计、阅读体验、SEO 和静态站点性能。系统还需要 B 端后台控制、Go Gin API、GORM、PostgreSQL、发布管理、评论审核、权限和审计。

## 方案

采用 Hybrid CMS 架构：

- PostgreSQL 存储后台编辑态内容、权限、配置、审计和发布记录。
- Go Gin API 负责后台业务规则和发布编排。
- Admin 后台负责内容生产和运营控制。
- 发布时生成 Hugo content 快照。
- Hugo + Stack 构建 C 端静态站点。
- C 端动态增强通过 Public API 异步加载。

## 备选方案

| 方案 | 优点 | 缺点 | 复杂度 |
|---|---|---|---|
| 纯 Hugo 静态博客 | 简单、快、SEO 好 | 没有成熟后台和数据库能力 | 低 |
| Gin 动态渲染全部页面 | 后台和前台数据一致 | 失去 Hugo/Stack 大量能力，需要重写主题 | 中高 |
| 前端 SPA + API | 动态能力强 | SEO 和阅读页首屏复杂度上升，Stack 复用困难 | 高 |
| Hybrid CMS | 保留 Stack，后台能力完整，发布可回滚 | 发布链路和双模型更复杂 | 中高 |

## 决策理由

Hybrid CMS 最符合用户目标：既要成熟后台，又要保留 Hugo 页面效果。它把“编辑态”和“发布态”分离，C 端性能和 SEO 不被后台系统拖累。

## 接受的权衡

- 接受数据库内容模型和 Hugo 内容模型之间的映射成本。
- 接受发布流水线复杂度。
- 接受初期先全量构建，后续再优化增量发布。

## 后果

正向：

- C 端稳定、快、可缓存。
- 后台具备完整 CMS 能力。
- 发布产物可追踪、可回滚。

负向：

- 发布链路需要 worker、快照、构建和回滚机制。
- 内容模型演进需要同步到 Hugo front matter。

缓解：

- 用 ADR 固定边界。
- 用 publish job 状态机管理发布。
- 用 manifest 记录每次发布快照。
