# Goal 执行与多 Agent 调控规范

## 1. 中心窗口机制

中心窗口负责：

- 维护总目标。
- 维护阶段计划。
- 分配子 agent 工作。
- 汇总工作日志。
- 决策冲突与优先级。
- 控制上下文膨胀。
- 在上下文接近上限时产出接力文档。

中心窗口保留：

- 当前目标。
- 已完成事项。
- 正在进行事项。
- 阻塞点。
- 决策记录。
- 关键文件与命令。
- 下一步任务。

## 2. 子 Agent 角色

- Coordinator：中心调度、计划、日志、验收。
- Architecture Agent：架构、ADR、边界。
- Backend Agent：Gin、GORM、PostgreSQL、API。
- Admin Agent：后台 UI、表单、状态管理。
- Hugo Agent：主题、内容生成、静态发布。
- Ops Agent：Docker、部署、CI/CD、备份。
- QA Agent：测试、验收、回归清单。
- Docs Agent：文档归档、runbook、接力总结。

## 3. 分配原则

- 每个子 agent 一次只拿明确任务。
- 每个任务必须有任务编号。
- 子 agent 输出必须包含：
  - 改动摘要。
  - 涉及文件。
  - 验证结果。
  - 阻塞点。
  - 下一步建议。
- 跨领域任务先由中心窗口拆分。
- 冲突由中心窗口裁决。
- 子 agent 不直接改变 roadmap，必须提交建议给中心窗口。
- 上下文过大时，先写工作日志，再开新窗口继续。

## 3.1 任务状态机

任务状态必须使用以下枚举：

- `Backlog`：已记录，尚未准备实施。
- `Ready`：依赖满足，可派发。
- `Assigned`：中心窗口已派给指定 agent。
- `In Progress`：agent 正在执行。
- `Review`：agent 已回报，等待中心窗口验收。
- `Blocked`：因依赖、决策、冲突或验证失败暂停。
- `Done`：中心窗口验收完成，日志已更新。
- `Superseded`：任务被新任务替代，不再执行。

任何任务进入 `In Progress` 前，必须登记 owner、允许编辑范围、禁止编辑范围和验证方式。

运行时状态维护在 `docs/process/task-board.md`。

## 3.2 中心窗口调度 SOP

每轮执行：

1. 阅读 `docs/process/task-board.md`。
2. 阅读 `docs/process/worklog.md` 顶部状态摘要和最近 3 条日志。
3. 运行 `git status --short --branch`。
4. 从 WBS 选择 `Ready` 任务。
5. 判断中心窗口本地执行和子 agent 并行执行的边界。
6. 登记任务状态和文件锁。
7. 派发子 agent。
8. 子 agent 回报后中心窗口验收。
9. 更新看板、工作日志和接力文档。

## 9. 协作纪律

- 先写计划，再拆任务，再实施。
- 每个阶段结束必须做阶段验收。
- 所有重要决策写入 ADR 或工作日志。
- 不在无任务编号的情况下做大改动。
- 不把临时方案沉淀为长期架构。
- 不让子 agent 静默工作，必须回传日志。
- 上下文接近上限时，优先产出接力记录。
