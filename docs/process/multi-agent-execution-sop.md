# 多 Agent 工程执行 SOP

本文把中心窗口与多个子 agent 的协作方式具体化。目标是让项目在长时间、多窗口、多 agent 的情况下仍然可控、可验收、可接力。

## 1. 角色定义

### Center

当前对话窗口，负责：

- 解释用户目标。
- 拆解 WBS。
- 维护 task board、worklog、handoff。
- 派发子 agent。
- 控制文件锁。
- 合并结果。
- 做最终验收。
- 向用户汇报。

### Explorer Agent

只读分析者，负责：

- 阅读代码或文档。
- 回答具体问题。
- 找缺口、风险、方案对比。
- 不直接修改文件。

### Worker Agent

有界执行者，负责：

- 在明确文件范围内实现代码或文档。
- 不修改未授权文件。
- 不回滚他人改动。
- 提供修改文件和验证结果。

### QA Agent

验证者，负责：

- 运行测试、构建、浏览器检查。
- 记录失败原因。
- 不修复非授权范围的问题。

## 2. 中心窗口每日循环

1. 读取 `docs/process/task-board.md`。
2. 读取 `docs/process/worklog.md` 顶部摘要和最近 3 条日志。
3. 运行 `git status --short --branch`。
4. 识别正在进行、阻塞、待验收任务。
5. 选择本轮任务。
6. 判断是否需要子 agent。
7. 登记任务状态和文件锁。
8. 派发任务。
9. 本地推进关键路径工作。
10. 收集 agent 汇报。
11. 复核 diff 和验证结果。
12. 更新文档、看板、日志。
13. 向用户汇报。

## 3. 派发任务模板

```markdown

任务名称：
背景：

你不是单独在代码库里工作，当前有中心窗口和其他 agent。不要回滚用户或其他 agent 的改动。

允许编辑：
禁止编辑：
只读资料：
输入假设：
预期产物：
验证方式：
完成后汇报：
- 完成状态
- 读取资料
- 修改文件
- 执行命令
- 验证结果
- 风险
- 阻塞
- 下一步建议
```

## 4. 子 Agent 回报验收清单

中心窗口收到回报后必须检查：

- 是否只改了允许范围。
- 是否有未说明的额外文件。
- 是否运行了约定验证。
- 是否更新了必要文档。
- 是否引入真实密钥或本地私有路径。
- 是否破坏既有用户改动。
- 是否符合阶段 DoD。

验收结论必须写入 `docs/process/worklog.md`。

## 5. 并行策略

适合并行：

- API 契约设计与 Admin 页面草图。
- 数据库模型设计与 Hugo front matter 映射。
- 运维 runbook 与安全基线。
- 测试策略与需求验收清单。
- 只读源码分析与本地文档编写。

不适合并行：

- 多个 agent 同时改同一 migration。
- 多个 agent 同时改同一个路由入口。
- 多个 agent 同时重构主题结构。
- 后台技术栈未确认前初始化 `apps/admin`。
- 数据模型未稳定前大规模写 Admin 表单。

## 6. 冲突处理流程

发现文件冲突：

1. 暂停相关任务状态为 `Blocked`。
2. 运行 `git status --short --branch`。
3. 确认冲突来源：用户、中心窗口、子 agent、生成产物。
4. 如果是用户改动，保留并围绕它继续。
5. 如果是子 agent 越界，拒收越界部分，必要时要求重做。
6. 由中心窗口手动整合，不让多个 agent 互相覆盖。
7. 工作日志记录冲突原因和处理结论。

## 7. 验证失败流程

验证失败时：

1. 任务进入 `Blocked` 或保持 `Review`。
2. 日志记录失败命令、失败摘要、涉及文件。
3. 中心窗口判断是实现错误、环境错误、测试错误还是文档过期。
4. 若问题局部且文件边界清晰，可派原 agent 修复。
5. 若涉及架构或需求变更，先更新 ADR/需求，再继续。

## 8. 上下文爆窗流程

触发条件：

- 对话上下文接近上限。
- 阶段完成，需要交接。
- 多个任务并行导致状态复杂。
- 中心窗口判断继续可能丢关键决策。

爆窗前必须完成：

- 更新 `docs/process/task-board.md`。
- 更新 `docs/process/worklog.md` 顶部状态摘要。
- 更新 `docs/process/context-handoff.md` 的当前状态、未完成、下一步。
- 记录 `git status --short --branch`。
- 说明哪些 agent 已完成、哪些未完成。
- 明确新窗口第一项任务编号。

## 9. 新窗口接手流程

新窗口必须：

1. 读取 `docs/process/context-handoff.md`。
2. 读取 `docs/process/worklog.md` 顶部摘要和最近 3 条日志。
3. 读取 `docs/process/task-board.md`。
4. 读取 `docs/plan/goal-plan.md` 和 `docs/plan/work-breakdown-structure.md`。
5. 运行 `git status --short --branch`。
6. 在首次汇报中说明接手的任务编号、状态、文件范围和下一步。
7. 不重复清空目录、不重新克隆、不覆盖当前仓库。

## 10. 工作日志追加规则

必须写日志的事件：

- 用户做出决策。
- 中心窗口派发任务。
- 子 agent 完成任务。
- 文件锁释放。
- 阶段验收。
- 验证失败。
- 架构或范围变化。
- 新窗口接力。

日志格式：

```markdown
## YYYY-MM-DD HH:mm +08:00 - <窗口/Agent> - <任务编号>

- 目标：
- 输入：
- 操作：
- 产出：
- 验证：
- 决策：
- 风险/问题：
- 下一步：
```

## 11. 用户汇报格式

普通进展汇报：

```markdown
当前阶段：
本轮完成：
验证结果：
风险：
下一步：
需要你决策：
```

阶段完成汇报：

```markdown
阶段：

验证命令：
未完成/风险：

```
